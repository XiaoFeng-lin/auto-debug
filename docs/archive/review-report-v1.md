## 一、文档类型判断

**auto_debug_product_spec.md** 属于 **PRD（产品需求文档） + 系统设计文档** 的混合体：

- **PRD 元素**：产品概述、用户输入规范、核心工作流、invocation 方式、目标平台
- **系统设计元素**：Agent 架构、子 Agent 调度规则、项目特定适配、记忆机制、进度追踪

### 这个文档目前的定位
这是**中高保真原型**——功能逻辑已完整，但缺乏：
- 用户故事与验收标准（AC）
- 技术实现的详细设计（接口、数据结构、算法复杂度）
- 测试策略（单元测试、集成测试、端到端测试）
- 错误处理与边界情况处理
- 监控与可观测性指标
- 性能指标（超时、上下文窗口占用）
- 兼容性矩阵（不同项目类型的覆盖率）
- 回滚与灾难恢复策略
- 安全审计清单

## 二、文档评审：发现的问题

### 2.1 关键缺失

| 缺失项 | 影响 | 建议 |
|--------|------|------|
| **用户故事与验收标准** | 无法验证功能是否满足预期 | 为每个主要用例（如内联日志、目录日志、smart-sync、sync-core-service）编写 Given-When-Then AC |
| **技术实现细节** | 开发时会发生歧义 | 补充：子 Agent 输入输出 JSON Schema、Task 命名规则、Memory 存储格式、上下文隔离阈值 |
| **错误处理策略** | 边界情况会掉坑 | 为每个阶段定义失败时的退出码、重试机制、降级方案（超时了怎么办？磁盘满怎么办？） |
| **超时控制** | 子 Agent 长时间运行会卡住 | 明确每个子 Agent 的最大执行时间（建议 120s），超时后的主 Agent 行为（continue / fallback / abort） |
| **上下文窗口管理** | 多文件 read 可能溢出 | 定义一次能 read 多少文件、文件大小上限、如何分批 |
| **配置化参数** | 调试困难 | 将上下文行数、超时、关键词列表、默认日志文件等做成可配置项 |
| **测试用例库** | 无法验证修复正确性 | 为三大 Bug 类型（逻辑缺陷、并发问题、遗漏场景）设计典型案例，包括已知根因和修复方法 |
| **版本兼容性矩阵** | Skill 发布后会面临不同版本的 Claude 模型 | 声明支持的最小模型版本（Sonnet 4+ / Opus 3+），以及不同模型的行为差异 |
| **安全审计清单** | 代码修改权限必须严格管控 | 明确哪些路径可以修改、哪些禁止，如何防止误删 |
| **输出文件命名冲突** | timestamp 不够精确可能冲突 | 建议使用更精确的 `auto_debug_extracted_{YYYYMMDD_HHMMSS_ffffff}.log` |
| **多进程并发安全** | 多个用户同时执行 Skill 会互相覆盖结果 | 临时文件应使用进程 ID 或随机 UUID |

### 2.2 模糊/冲突定义

| 问题 | 说明 | 建议 |
|------|------|------|
| "子 Agent 失败时主 Agent 接管" | 但没有定义什么是失败、接管的具体步骤 | 明确失败条件：超时、工具异常、空结果、未预期的输出格式；接管动作：重试 / 转为主 Agent 同步执行 / 询问用户 |
| "子 Agent 只返回结构化结论" | 结论的 schema 未定义 | 设计子 Agent 输出 schema（至少包含：`root_cause`, `evidence` 列表, `suggested_fix`, `confidence`, `files_analyzed`） |
| "大量日志污染主上下文" | 何为"大量"？ | 定义阈值：子 Agent 返回的提取结果应 < 100KB；超过则聚合摘要 |
| "用户提供的日志路径" | 如果路径不存在或不可写怎么办？ | 阶段 2 环境检查应包含：路径存在性、可读性、可写性（用于输出） |
| "暂停并询问用户" | 询问什么？回复格式？超时？ | 明确用户等待时间（建议 60s）、默认回复（abort）、多次询问后行为（fallback） |
| "不执行可能产生副作用的命令" | list files 算副作用吗？git branch 呢？ | 给出的允许命令白名单、禁止命令黑名单 |
| "置信度评分 X/100" | 评分算法是什么？ | 定义评分因子及权重（日志证据 40%, 代码确定性 35%, 复现可能性 25%） |
| "临近的上下文区间重叠时合并" | 重叠多少算重叠？ | 定义：两段提取的起始行差 < 2 × 上下文行数时合并 |
| "commit 不在当前分支 → 执行 git branch -r --contains" | 但可能不在本地已拉取的分支中，需要 git fetch 吗？ | 明确：只查本地已拉取的分支；未命中则提示用户可能来自已删除分支或未 fetch 的分支 |

### 2.3 可落地性问题

1. **"Grep-first" 策略依赖 ripgrep/grep** —— 需要假设这些工具可用且路径正确（Windows/macOS/Linux 都要验证）
2. **子 Agent 的并行度** —— 是否允许并行运行 Stage 3 和 Stage 4？文档未说明调度策略
3. **子 Agent 的输出大小限制** —— 未说明接口数据大小限制（Claude Code 可能有 payload 限制）
4. **Skill invocation 语法** —— 文档给出了 `/auto-debug` 但 Skill 需在 `skill.md` 中声明 `triggers` 模式
5. **CLAUDE.md 中的配置化需求** —— 文档提到"应支持配置化"，但未说明配置存储位置（项目根目录的 `.auto-debug-config.json`？还是 CLI 参数？）

### 2.4 边界情况未覆盖

| 场景 | 风险 | 建议处理 |
|------|------|----------|
| 日志在轮转过程中，部分 .1 文件被系统删除 | 文件不存在时跳过并记录警告 | 捕获 FileNotFoundError，继续处理其他文件 |
| 日志文件损坏（不完整、乱码） | 解析失败 | try-catch，跳过并记录，不影响其他文件 |
| 用户提供的 Bug 描述太宽泛（如"程序崩溃"） | 关键词检索返回海量结果，上下文爆炸 | 阶段 1 检测关键词稀疏度，若匹配行 > 1000 行则询问用户补充信息 |
| 日志中找不到 commit hash / version 标识 | 无法执行分支比对，信任度降低 | 在报告中明确标注"无法识别版本信息"，置信度 -20 分 |
| 代码分析时发现多个潜在根因 | 混淆用户 | 列出所有可能性，附带各自的置信度，供用户选择 |
| 修复方案涉及多个文件的多处修改 | 执行复杂，容易出错 | 提供修改清单，逐步执行，每步验证 |
| 修复后代码无法通过 lint/test | 修复失败 | 回滚至原始状态，输出错误报告，要求用户介入 |

## 三、标准项目开发流程与必备文档

### 3.1 从 PRD 到上线的典型流程（适用于跨平台 Skill）

```
1️⃣ PRD（产品需求文档）
   ├─ 产品愿景、目标用户、核心价值
   ├─ 用户故事（User Stories）与 AC（Acceptance Criteria）
   └─ 成功指标（Success Metrics）

2️⃣ 技术设计文档（Technical Design / TDD）
   ├─ 架构图（子 Agent 通信、数据流）
   ├─ API / 接口定义（Input Schema / Output Schema）
   ├─ 数据模型（配置、缓存、报告格式）
   ├─ 错误处理与状态机
   ├─ 性能与规模预估（日志吞吐量、最大文件大小、超时）
   └─ 安全与权限模型

3️⃣ 实现计划（Implementation Plan）
   ├─ 开发 Milestone 拆分（如：MVP、Pilot、GA）
   ├─ 每个 Milestone 的详细任务清单（含测试）
   ├─ 风险与缓解措施
   └─ 依赖项清单

4️⃣ 测试计划（Test Plan）
   ├─ 单元测试（针对各解析器、分支比对逻辑、评分算法）
   ├─ 集成测试（主 Agent + 子 Agent 串通）
   ├─ 端到端测试（e2e 场景：smart-sync 全流程、sync-core-service 全流程）
   ├─ 边界测试（大文件、无权限、损坏日志、超时模拟）
   └─ 人工验收测试用例（UAT）

5️⃣ 部署与运维文档（Runbooks）
   ├─ 安装与配置指南
   ├─ 常见问题（FAQ）与故障排除
   ├─ 监控指标（执行计数、成功率、平均耗时、上下文命中率）
   ├─ 日志规范（Skill 自身的调试日志输出到哪里）
   └─ 升级与版本管理（Skill version bump）

6️⃣ 用户文档（User Guide）
   ├─ 快速开始
   ├─ Invocation 语法详述（参数说明、示例）
   ├─ 输入规范（日志要求、Bug 描述要求）
   ├─ 输出解读（报告结构、置信度含义）
   └─ 安全与隐私提醒

7️⃣ 法律与合规（如必要）
   ├─ 隐私政策（日志是否外传？内存存储？）
   └─ 开源许可证（如果涉及第三方库）
```

## 四、技术可行性分析：是否需要辅助脚本？

### 4.1 纯 Skill 能力的限制

Claude Code Skill 本质上是一个 **markdown 文件 + 工具调用指令**：

- ✅ **可以**：调用 Read、Grep、Bash、TaskCreate、Agent 等工具
- ❌ **无法**：安装 Python 包、注册全局快捷键、修改 settings.json 自动生效（除非用户主动执行）
- ⚠️ **限制**：Tool use payload 大小限制（约几百 KB）、子 Agent 结果需序列化回主 Agent

### 4.2 日志提取的复杂度评估

| 任务 | 纯 Skill 可实现性 | 风险 |
|------|-------------------|------|
| 读取单个 50MB 文件并 grep | ✅ 可用 `Grep` 工具（支持 regex） | Grep 不支持时间范围过滤，需先过滤后解析 |
| 读取多个轮转文件（debug.log ~ debug.log.10） | ✅ 可用 `Glob` + `Grep` | 需要多次工具调用，上下文膨胀风险 |
| 解析 commit hash 并从 git 历史查找分支 | ✅ `Bash - git branch -r --contains <hash>` | 但输出解析需要复杂的 prompt 工程 |
| 合并重叠的上下文区间 | ✅ 在主 Agent 阶段进行 | 若提取行太多（>10,000）会溢出上下文 |
| 写入 `auto_debug_extracted_*.log` | ✅ `Write` 工具 | 无风险 |

**最关键的瓶颈**是：
1. **Grep 不支持时间过滤** → 需要在主 Agent 读取匹配行后再根据时间过滤（这又会读入大量行）
2. **日志量极大时的上下文管理** → 需要严格的"结果聚合"逻辑，不能把每个匹配行的全部上下文都传回
3. **子 Agent 的输出大小限制** → 若 log 有 1000 处匹配，×10 行上下文 = ~30KB，还能接受；若是 10000 处，就爆炸了

### 4.3 是否引入辅助脚本的建议

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **纯 Skill（无脚本）** | 部署简单、用户零配置、可立即使用 | 复杂逻辑（时间过滤、区间合并）需在 prompt 中硬编码，难维护；性能受限（多次工具调用）；调试困难 | ⚠️ **仅适合 MVP / 验证期**，规模大了会碰到上下文上限 |
| **Skill + Prebuilt Script**（如 Python 脚本，先 git clone 到项目）| 复杂逻辑封装为函数，可单元测试；性能高（本地计算）；降低 Skill prompt 复杂度 | 需要用户运行安装脚本；脚本版本与 Skill 需同步更新；跨平台兼容问题 | ✅ **Pilot / GA 推荐方案** |
| **Skill + Runtime Script Generation**（动态生成并执行）| 无需预装依赖；灵活性高 | 每次运行需要生成脚本→执行→清理，有安全风险；脚本内容依赖 LLM 生成，可重复性差 | ❌ **不推荐** |

**推荐路径**：
1. **阶段 1（2-4 周）**：纯 Skill MVP，验证核心工作流，性能阈值设定为：日志文件总数 ≤ 50 个，单文件 ≤ 100MB，匹配行数 ≤ 500 行
2. **阶段 2（4-6 周）**：如果用户反馈性能或复杂度问题 → 引入 Python 辅助脚本（如 `auto_debug_extractor.py`，负责时间过滤、区间合并、commit 比对），主 Skill 只负责调用和执行
3. **阶段 3**：封装为可发布 package（`pip install auto-debug-cli`），用户只需 `auto-debug` 命令，内部自动启动 Claude Code Skill 或 standalone 模式

## 五、距离正式开发还需要做的具体事项

### 立即执行（开发前）

1. **确认文档类型与基础问题**（需要 1-2 小时）
   - ✅ 产品定位：已确认（Claude Code + Codex，内部使用 → 开源）
   - ✅ 是否用脚本：建议"先纯 Skill MVP，评估后引入"
   - 剩下：确定 **成功指标**（如：诊断准确率 > 80%，平均耗时 < 5 分钟，用户满意度 > 4/5）

2. **补充 PRD 缺失内容**（需要 4-8 小时）
   - 用户故事与 AC（3-5 个核心用例）
   - 数据字典（日志路径、时间范围、关键词的数据类型、约束、示例）
   - 错误代码表（每个阶段的退出码、错误消息模板）
   - 配置清单（超时、上下文行数、默认日志文件列表）

3. **技术设计定稿**（需要 2-3 天）
   - 接口定义（Input Schema、Output Schema 的 JSON Schema）
   - 状态机（7 阶段的切换条件）
   - 子 Agent 的 prompt 模板（需要用变量占位符，如 `{bug_description}`, `{log_path}`）
   - 数据结构设计（结果聚合、区间合并算法）
   - 性能模型（GB 级日志需要多少时间）

### 开发阶段（按 Sprint 规划）

**Sprint 0：工具链准备（1 天）**
- 项目初始化（git repo、gitignore、requirements.txt 或 package.json）
- 工具选择：是否用 `task`、`typer`、`rich` 等（如用脚本）
- 配置管理：env 变量、config 文件结构

**Sprint 1：MVP 实现（1 周）**
- Stage 1-2（输入校验 & 环境检查） — 2 天
- Stage 3（日志提取） — 2 天
- Stage 4（代码分析） — 2 天
- Stage 5（报告生成） — 1 天

**Sprint 2：人工确认与修复执行（1 周）**
- 用户交互（确认、选择方案）
- Stage 6（代码修复） — 2 天
- Stage 7（修复验证） — 2 天
- 整合测试、文档编写 — 2 天

**Sprint 3：优化与测试（1 周）**
- 单元测试覆盖率 > 80%
- 性能优化（上下文压缩、并发）
- 边界测试
- Runbook 与用户文档定稿
