# Auto-Debug 产品功能规格说明书

> 版本：1.1
> 更新日期：2026-04-30
> 状态：MVP 开发完成

---

## 一、产品概述

### 1.1 定义
基于用户提供的 Bug 描述与日志，自主分析定位问题根因，输出诊断报告与修复方案，经用户确认后执行代码修复的自动化工具。

### 1.2 产品形态
**Skill** — 具有通用性，可运行于 Claude Code、Codex 及其他支持 Skill 机制的 Agent/插件平台。

### 1.3 产品名称
`auto-debug`

### 1.4 核心原则

| 原则 | 说明 |
|------|------|
| **不猜测** | 任何不确定之处必须与用户确认，严禁假设或推测 |
| **不擅自写入** | 分析阶段仅读取代码，修复阶段需经用户确认后才写入 |
| **全程可追踪** | 每个阶段的状态和结论都有记录，用户可随时审查 |
| **上下文隔离** | 大量日志处理交由子 Agent 执行，避免污染主 Agent 上下文 |
| **项目强绑定** | 必须在项目根目录下执行，确保代码上下文可用 |

## 二、用户输入规范

### 2.1 必需输入

| 字段 | 说明 | 示例 |
|------|------|------|
| **Bug 描述** | 现象、预期行为、实际行为的说明 | "同步文件夹时，重命名操作后旧文件未删除" |
| **日志来源** | 内联日志内容 或 日志路径 | 见下方 2.2 / 2.3 |

### 2.2 日志输入方式一：内联粘贴
用户直接在描述中粘贴关键日志片段。适用于日志量少、用户已自行筛选的场景。

```
/auto-debug
Bug: 同步文件夹时重命名后旧文件残留
日志：
[2026-04-29 10:23:45] ERROR sync_handler.py:142 - Rename operation failed: target already exists
[2026-04-29 10:23:45] WARNING sync_handler.py:150 - Skipping cleanup, lock held by process 3821
```

### 2.3 日志输入方式二：提供路径
用户指定日志文件或日志目录路径。适用于日志量大、需系统自动筛选的场景。

```
/auto-debug
Bug: 同步文件夹时重命名后旧文件残留
日志路径: D:\bug_log\smart_sync\Fangcloud\SmartSync
```

**路径类型判定规则：**

| 路径类型 | 判定条件 | 处理方式 |
|----------|----------|----------|
| **文件路径** | 路径指向单个文件 | 仅在该文件内提取关键信息 |
| **目录路径** | 路径指向目录 | 多文件检索与筛选（见 3.2 节） |

### 2.4 可选输入

| 字段 | 说明 | 默认行为 |
|------|------|----------|
| 时间范围 | Bug 发生的大致时间 | 检索全部日志 |
| 关键词/文件名 | 用户提及的出错文件或模块名 | 无预设过滤 |
| 环境信息 | 运行时版本、操作系统等 | 自动从项目中检测 |
| 项目特定标记 | 如 commit_hash 等 | 自动检测项目类型后决定是否启用 |

## 三、核心工作流

### 3.1 总体流程

```
[输入] 用户描述 + 日志
        ↓
[阶段1] 输入校验与歧义澄清 ←── 如有歧义，暂停并询问用户
        ↓
[阶段2] 环境检查（是否在项目目录下）
        ↓
[阶段3] 日志提取与清洗（子 Agent 执行）
        ↓
[阶段4] 代码分析与根因定位（子 Agent 执行）
        ↓
[阶段5] 诊断报告生成 + 修复方案输出
        ↓
[确认] 用户审核并决策 ←── 需人工介入
        ↓
[阶段6] 代码修复执行
        ↓
[阶段7] 修复验证
        ↓
[输出] 最终结果
```

### 3.2 阶段详细说明

#### 阶段1：输入校验与歧义澄清

**职责**：解析用户输入，确认理解无误。

**规则**：
- 提取 Bug 描述中的关键实体：出错模块/文件、错误类型、预期 vs 实际行为
- 识别日志输入方式（内联 vs 路径）
- 对以下情况**必须暂停并询问用户**：
  - Bug 描述含糊，无法确定问题边界
  - 日志路径不存在或不可访问
  - 日志内联内容与描述不匹配
  - 用户描述存在矛盾信息

**输出**：结构化的 Bug 信息摘要，供后续阶段使用。

---

#### 阶段2：环境检查

**职责**：确认执行环境满足要求。

**检查项**：

| 检查项 | 条件 | 不满足时的行为 |
|--------|------|----------------|
| 项目目录 | 当前工作目录是有效的项目根目录（包含项目标识文件，如 .git、package.json、pom.xml 等） | **暂停**，通知用户切换到项目目录后重新执行 |
| 代码可读 | 项目代码文件可正常读取 | **暂停**，提示权限问题 |
| 日志路径可达 | 用户提供的日志路径存在且可读 | **暂停**，提示路径问题 |

---

#### 阶段3：日志提取与清洗

**职责**：从日志源中提取与当前 Bug 相关的关键信息。

**执行方式**：由**子 Agent** 执行，结果返回主 Agent。

##### 3.3.1 内联日志处理
用户已粘贴关键日志 → 直接结构化，无需额外提取。

##### 3.3.2 单文件日志处理
路径为文件 → 读取该文件 → 根据用户描述的关键词/时间范围进行过滤 → 提取相关上下文。

##### 3.3.3 多文件日志处理（目录路径）
路径为目录 → 需执行以下子步骤：

**步骤 A — 日志文件发现**
- 扫描目录下所有日志文件
- 按修改时间排序，优先分析近期文件
- 识别日志文件格式（.log、.txt 等）

**步骤 B — 智能检索与过滤**
- 基于 Bug 描述中的关键词检索（文件名、模块名、错误类型等）
- 基于时间范围过滤（如用户提供）
- 基于 ERROR/WARNING 等日志级别过滤
- 提取匹配行的上下文（前后各 N 行，N 可配置）

**步骤 C — 信息提取**
- 提取错误堆栈信息
- 提取时间线中的关键事件序列
- 提取相关变量值和状态信息

**步骤 D — 项目特定增强**
- **smart-sync / sync-core-service 项目**：
  - 从日志中提取 `commit_hash` / `commit_id`
  - 与当前项目分支的 commit 历史比对
  - 若 commit 不在当前分支 → 标注该日志对应的分支名称
- **其他项目**：跳过 commit 比对步骤

**步骤 E — 结果输出**
- 将提取的关键日志信息写入新文件，保存在用户提供的日志路径下
- 文件命名规范：`auto_debug_extracted_{timestamp}.log`
- 将结构化的关键信息返回主 Agent

---

#### 阶段4：代码分析与根因定位

**职责**：结合日志与源代码，定位 Bug 根因。

**执行方式**：由**子 Agent** 执行，结果返回主 Agent。

**权限**：仅读取代码，**不写入任何文件**。

**分析流程**：

```
日志中的错误信息
        ↓
定位出错代码位置（文件:行号）
        ↓
读取出错代码及其上下文
        ↓
分析调用链（向上追溯调用者，向下查看被调用者）
        ↓
分析数据流（追踪关键变量的来源与去向）
        ↓
识别根因模式（逻辑缺陷 / 遗漏场景 / 并发问题 / 配置错误 / ...）
        ↓
验证分析结论（尝试在代码中找到支持/反驳的证据）
        ↓
若存在不确定性 → 暂停，向用户确认
```

**Bug 类型覆盖范围（第一版即全面支持）**：

| 类别 | 典型示例 |
|------|----------|
| 逻辑缺陷 | 条件判断错误、循环边界错误、状态机转换遗漏 |
| 遗漏场景 | 未处理的边界条件、空值未处理、异常未捕获 |
| 并发问题 | 竞态条件、死锁、资源未释放 |
| 数据问题 | 数据格式不匹配、编码问题、序列化/反序列化错误 |
| 配置错误 | 环境变量缺失、配置项冲突 |
| 集成问题 | API 调用方式错误、接口参数不匹配 |
| 语法/类型错误 | 新建项目场景下可能出现（已考虑） |
| 性能问题 | 内存泄漏、无限循环、资源耗尽 |
| 其他 | 不属于以上分类的 Bug |

**分析不确定时的处理**：
- 分析出多个可能的根因 → 全部列出，标注各自的证据和概率
- 无法确认根因 → 明确告知用户当前分析的瓶颈，请求更多信息
- **严禁**：跳过不确定性直接给出结论

---

#### 阶段5：诊断报告生成与修复方案输出

**职责**：将分析结果组织为标准化报告，生成修复方案。

##### 5.5.1 诊断报告内容

```markdown
# Bug 诊断报告

## 基本信息
- 生成时间：{timestamp}
- 项目：{项目名称}
- 当前分支：{git_branch}
- Bug 描述：{用户原始描述}

## 问题摘要
{一句话总结根因}

## 根因分析
{详细的技术分析，包含：}
- 出错代码位置（文件:行号，附带代码片段）
- 错误触发条件
- 数据流/控制流分析
- 根因分类（逻辑缺陷/遗漏场景/...）

## 影响范围
- 直接影响：{哪些功能/模块受影响}
- 潜在影响：{修复可能引起的连锁影响}

## 复现步骤
1. {前置条件}
2. {操作步骤}
3. {预期结果 vs 实际结果}

（注：如无法确定完整复现步骤，标注"部分推断"，并说明依据）

## 修复方案
### 方案描述
{修复思路说明}

### 代码变更
```diff
{具体的 diff 内容，包含上下文}
```

### 修改文件列表
- `{文件路径}`: {修改说明}

## 置信度评分
**{X}/100**

评分依据：
- 日志证据充分度：{高/中/低}
- 代码分析确定度：{高/中/低}
- 复现可能性：{高/中/低}

## 项目特定信息
{如适用：commit 比对结果、分支信息等}
```

##### 5.5.2 报告输出
- 在终端/对话中展示完整报告
- 同时写入文件：`auto_debug_report_{timestamp}.md`，保存在用户提供的日志路径下

---

#### 确认环节（人工介入）

**职责**：用户审核诊断报告与修复方案。

**交互方式**：
- 展示报告后，等待用户决策
- 用户可选操作：

| 操作 | 说明 |
|------|------|
| **确认修复** | 批准方案，进入阶段6执行代码修复 |
| **修改方案** | 用户提出调整意见，重新生成修复方案 |
| **仅采纳分析** | 认可诊断但暂不修复 |
| **否决** | 不认可分析结果，流程终止 |

---

#### 阶段6：代码修复执行

**职责**：按用户确认的修复方案修改代码。

**前置条件**：用户已明确确认修复。

**执行规则**：
- 严格按照诊断报告中的修复方案执行
- 每处修改使用 Edit 工具，保留修改记录
- 如修复过程中发现新问题（修复方案与实际代码不符等），**暂停并告知用户**，不得擅自调整方案

---

#### 阶段7：修复验证

**职责**：验证修复的正确性。

**验证项**（按项目实际情况选择可执行的）：

| 验证项 | 方式 |
|--------|------|
| 语法检查 | 编译/解释器检查 |
| 测试执行 | 运行项目测试套件 |
| 静态分析 | Lint / 类型检查 |
| 回归检查 | 确认修复未引入新问题 |

**验证结果处理**：
- 全部通过 → 输出成功报告
- 存在失败 → 输出失败详情，提示用户决定后续操作（回滚 / 继续修复 / 终止）

## 四、Agent 架构

### 4.1 角色分工

```
主 Agent (Coordinator)
  │
  ├─ 子 Agent A: 日志提取与清洗
  │    └─ 输入: 日志路径 + Bug 描述关键词
  │    └─ 输出: 结构化关键日志信息 + 提取结果文件
  │
  ├─ 子 Agent B: 代码分析与根因定位
  │    └─ 输入: 关键日志信息 + 项目代码
  │    └─ 输出: 根因分析结论 + 修复建议
  │
  └─ 主 Agent: 协调调度 + 报告生成 + 修复执行
```

### 4.2 子 Agent 调度规则

| 规则 | 说明 |
|------|------|
| 日志提取必须走子 Agent | 避免大量日志内容污染主上下文 |
| 代码分析走子 Agent | 读取多个文件时避免上下文膨胀 |
| 子 Agent 只返回结构化结论 | 返回精炼的分析结果，而非原始内容 |
| 子 Agent 失败时主 Agent 接管 | 子 Agent 超时或异常，主 Agent 评估是否自行处理或暂停询问用户 |

### 4.3 进度追踪

使用 TaskCreate/TaskUpdate 跟踪各阶段状态，用户可随时查看进度：

| Task | 状态流转 |
|------|----------|
| 输入校验 | pending → in_progress → completed |
| 环境检查 | pending → in_progress → completed |
| 日志提取与清洗 | pending → in_progress → completed |
| 代码分析与根因定位 | pending → in_progress → completed |
| 诊断报告生成 | pending → in_progress → completed |
| 用户确认 | pending → in_progress → completed |
| 代码修复 | pending → in_progress → completed |
| 修复验证 | pending → in_progress → completed |

## 五、项目特定适配

### 5.1 smart-sync

#### 5.1.1 项目识别

**识别特征**（满足以下条件即识别为 smart-sync）：

| 序号 | 特征 | 说明 |
|------|------|------|
| 1 | 根目录包含 `sync/` 目录 | 核心代码所在目录 |
| 2 | 根目录包含 `build.py` | 打包构建脚本 |
| 3 | 根目录包含 `requirements.txt` | 依赖声明文件，内容含 `aiohttp`、`SQLAlchemy` 等特征依赖 |
| 4 | `sync/` 目录下存在 `smart_sync.py` | 主入口文件 |

> 注：`bug_log.txt` 不是项目文件，不作为识别依据。

#### 5.1.2 项目技术栈

| 项 | 值 |
|----|-----|
| 语言 | Python |
| 运行时要求 | Python 3.6.8+ |
| 打包方式 | PyInstaller (`build.py`) |
| 异步框架 | asyncio + aiohttp + tornado |
| 数据库 | SQLAlchemy |
| 错误上报 | Sentry SDK |
| 支持平台 | Windows / macOS |

#### 5.1.3 分支体系

**主要生产分支**：

| 分支名 | 用途 | 备注 |
|--------|------|------|
| `production_hubble_gray` | 同步分支（常用） | 本地已拉取 |
| `production_hubble_scatter` | 散落同步分支 | 基于 gray 扩展，公有云主要使用 |
| `production` | 公有云测试/生产包 | 远程分支 |
| `private_cloud` | 专有云生产包 | 远程分支 |

**分支关系**：
- `scatter` 版本是 `gray` 版本的扩展功能版本
- 排查 Bug 时需要确认日志对应的分支版本

#### 5.1.4 日志体系

##### 日志文件清单

| 日志文件 | Logger 名称 | 级别 | 最大备份文件数 | 单文件上限 | 最大总容量 |
|----------|-------------|------|---------------|-----------|-----------|
| `debug.log` | `sync.core` | DEBUG | 10 | 50MB | ~550MB |
| `error.log` | `sync.core` | ERROR | 2 | 50MB | ~150MB |
| `monitor.log` | `sync.monitor` | DEBUG | 10 | 50MB | ~550MB |
| `conflict.log` | `sync.conflict` | DEBUG | 2 | 50MB | ~150MB |
| `alive.log` | `sync.alive` | DEBUG | 1 | 50MB | ~100MB |

**日志文件命名规则**（Python RotatingFileHandler）：
- 当前写入文件：`debug.log`
- 轮转后：`debug.log` → `debug.log.1`，新内容写入新的 `debug.log`
- 最多保留 N 个备份文件（N = backupCount）

**日志默认存储路径**：
- `{user_data_dir}/FangCloud/SmartSync/` （Windows: `%APPDATA%/FangCloud/SmartSync/`，macOS: `~/Library/Application Support/FangCloud/SmartSync/`）

**用户提供的日志路径示例**：`D:\bug_log\smart_sync\Fangcloud\SmartSync`

##### 日志格式

```
{时间} - {进程ID} - {线程ID} - {日志级别} - {文件名}[line:{行号}] - {消息内容}
```

**实例**：
```
2026-04-28 14:26:31,044 - 55448 - 35324 - INFO - smart_sync.py[line:66] - meta: {"commit_hash": "557414e43f44211d7276bc5e537bba4af7c91e29"}
```

##### Commit Hash 比对机制

**提取方式**：从 `debug.log` 中检索以下格式的行：
```
smart_sync.py[line:66] - meta: {"commit_hash": "<hash>"}
```

**比对流程**：

```
从 debug.log 中提取 commit_hash
        ↓
执行 git rev-parse HEAD（获取当前分支最新 commit）
        ↓
   ┌─ 一致 → 日志来自当前分支，正常处理
   │
   └─ 不一致 → 执行 git branch -r --contains <hash>
               ↓
          ┌─ 匹配到远程分支 → 在报告中标注：日志来自 <branch_name> 分支
          │
          └─ 未匹配到任何分支 → 在报告中标注：日志对应的 commit 不在已知分支中，
                                可能来自已删除的分支或已 rebase 的提交
```

**特别处理**：
- 如果日志中的 commit 属于 `production_hubble_scatter`，报告中需标注"此日志来自散落同步版本"
- 如果日志中的 commit 属于 `production_hubble_gray`，标注"此日志来自普通同步版本"

##### 日志提取策略

**优先级**：`debug.log` > `monitor.log` > `error.log` > `conflict.log` > `alive.log`

**用户提及的主要排查日志**：`debug.log` 和 `monitor.log`

**提取方法**（应对总计可能超过 1GB 的日志量）：

采用 **Grep-first（搜索优先）** 策略，绝不全量读取日志文件：

```
步骤1 — 确定搜索范围
  ├─ 用户指定文件名 → 仅搜索匹配的日志文件
  └─ 用户未指定 → 默认搜索 debug.log* 和 monitor.log*

步骤2 — 提取 commit hash
  ├─ 从 debug.log（仅当前文件即可，启动日志在最新的 debug.log 中）
  └─ 执行 commit 比对流程

步骤3 — 关键词检索
  ├─ 基于 Bug 描述提取搜索关键词（文件名、模块名、错误类型等）
  ├─ 使用 ripgrep/grep 跨文件搜索匹配行
  ├─ 时间范围过滤（如用户提供）
  └─ 日志级别过滤（ERROR / WARNING 优先）

步骤4 — 上下文提取
  ├─ 对每个匹配行，提取前后各 5 行上下文（可配置）
  └─ 去重：相邻的上下文区间重叠时合并

步骤5 — 结果整理
  ├─ 按时间排序
  ├─ 去除冗余信息
  └─ 写入 auto_debug_extracted_{timestamp}.log 至用户指定路径
```

**为什么不全量读取**：
- debug.log 全系列可达 550MB，monitor.log 同理
- 子 Agent 的上下文窗口有限，全量读取会丢失信息
- Grep 可以在数秒内扫描 550MB 文件，精确命中相关行

#### 5.2.4 日志体系

##### 日志文件清单

| 日志文件 | 默认路径 | 日志库 | 最大文件大小 | 最大备份数 | 日志级别控制 |
|----------|----------|--------|--------------|-----------|-------------|
| `node_addon.log` | `{appdata}/FangCloud/{app_name}/` | easylogging++ | 50MB | 10 | `logging.conf` 配置文件控制 |

**日志文件命名规则**：
- 当前文件：`node_addon.log`
- 轮转后：`node_addon.log` → `node_addon.log.1` → `node_addon.log.2` ... 最旧版本为 `.10`
- 超过 50MB 触发轮转（在 `rolloutHandler` 中实现）

**日志格式**：
```
{时间} - {进程ID} - {线程ID} - {日志级别} - {文件名}[line:{行号}] - {消息内容}
```

**示例**：
```
2026-04-29 14:26:31,044 - 55448 - 35324 - INFO - transfer_manager.cpp[line:121] - init addon, commit_id: 3e80ae3
```

##### Commit ID / Version 比对机制

**日志中可能出现的两种版本标识**：

| 版本类型 | 日志格式 | 示例 | 适用范围 |
|----------|----------|------|----------|
| **新版本**（commit_id） | `init addon, commit_id: {8位短hash}` | `init addon, commit_id: 3e80ae3` | 最新版本 |
| **老版本**（手动版本号） | `init addon, version:{gAddonVersion}` | `init addon, version:2.0.6` | 大量存量用户仍在使用 |

**提取逻辑**：

```
从日志中检索启动行（node_addon 初始化阶段）
        ↓
   ┌─ 匹配到 "commit_id:" → 新版本格式
   │   提取 8 位短 hash
   │   比对方式：git rev-parse HEAD → 截取前8位 → 比对
   │   ┌─ 一致 → 日志来自当前分支
   │   └─ 不一致 → git branch -r --contains <hash> → 标注对应分支
   │
   └─ 匹配到 "version:" → 老版本格式
       提取手动版本号字符串
       比对方式：在 git log 中搜索对应 tag 或 commit message
       ┌─ 找到匹配的 tag（如 v2.0.6）→ 标注日志来自该 tag 对应的版本
       └─ 无法匹配 → 在报告中标注"版本号为手动维护，无法自动关联到具体 commit，
                      请用户确认日志对应的代码版本"
```

**特别处理**：
- 若日志中同时缺失两种版本标识 → 在报告中提示"日志中未找到版本信息，可能来自极早期版本或日志被截断"
- 老版本号无法精确映射到 commit 时，**不得猜测**，必须向用户确认

**与 smart-sync 的区别**：
- commit ID 是预编译宏，在启动时一次性记录到日志
- 因此 commit ID 总在日志的**最前面部分**（addon 初始化阶段）
- 不需要像 smart-sync 那样等待 meta.json 注入

##### 日志级别控制

通过 `logging.conf` 配置文件控制（放置于亿方云 app 安装目录下的 `resources/add_ons/` 目录）：

```ini
# DEBUG=TRUE 启用调试日志
# DEBUG=FALSE 禁用调试日志
# 不放置此配置文件 = DEBUG=FALSE
DEBUG=TRUE
```

**影响**：若用户环境中未启用 DEBUG 日志，排查信息可能不足。诊断报告中应提示此情况。

##### 日志提取策略

**优先级**：`node_addon.log`（当前文件优先，因为 commit ID 一定在启动日志中）

**用户提供的日志路径示例**：
- 单文件：`D:\bug_log\sync_core_service\FangCloud\{app_name}\node_addon.log`
- 目录：`D:\bug_log\sync_core_service\FangCloud\{app_name}\`

**提取方法**：与 smart-sync 相同，采用 **Grep-first** 策略：

```
步骤1 — 提取版本标识
  ├─ 从 node_addon.log 提取 commit_id 或 version
  └─ 执行版本比对流程

步骤2 — 关键词检索
  ├─ 基于 Bug 描述提取搜索关键词
  ├─ 使用 ripgrep 跨文件搜索（node_addon.log 及 .1 ~ .10）
  └─ 日志级别过滤（ERROR / WARNING 优先）

步骤3 — 上下文提取与去重

步骤4 — 结果整理并写入 auto_debug_extracted_{timestamp}.log
```

## 十一、用户故事与验收标准

### US-01：内联日志诊断

**作为** 开发者，**我想要** 提供简短的 Bug 描述和内联日志片段来快速诊断问题，**以便** 在日志量少、已自行筛选的场景下获得即时分析。

**验收标准**：

- **AC-01-1**：Given 用户提供了 Bug 描述和内联日志，When 启动 auto-debug，Then 直接进入日志结构化处理（跳过文件发现和检索步骤）
- **AC-01-2**：Given 内联日志内容与 Bug 描述不匹配（如描述提及"重命名"但日志中无任何重命名相关记录），When 进入阶段1，Then 暂停并提示用户确认日志是否正确
- **AC-01-3**：Given 内联日志包含完整的错误堆栈信息，When 完成诊断，Then 报告中必须包含该堆栈对应的源代码位置（文件:行号）
- **AC-01-4**：Given 内联日志内容为空或仅含无意义字符，When 进入阶段1，Then 暂停并提示用户补充有效日志

- **AC-02-2**：Given 日志文件路径不存在，When 进入阶段2环境检查，Then 暂停并提示"日志路径不存在，请确认路径后重新执行"
- **AC-02-3**：Given 日志文件存在但无读取权限，When 进入阶段2环境检查，Then 暂停并提示权限问题

- **AC-03-2**：Given 目录下日志文件总数 > 50 个，When 进入阶段3步骤A，Then 按修改时间排序后仅分析最近 50 个文件，并在报告中标注"仅分析了最近的 50 个日志文件"
- **AC-03-3**：Given 多文件检索后总匹配行数 > 1000 行，When 进入阶段3步骤C，Then 子 Agent 对结果进行聚合摘要（按时间线归纳关键事件，而非返回全部原始行），确保返回主 Agent 的结构化数据 < 100KB
- **AC-03-4**：Given 某个轮转日志文件（如 debug.log.3）在检索期间被系统删除，When 子 Agent 尝试读取，Then 跳过该文件并继续处理其他文件，在提取结果中记录"文件 debug.log.3 已不存在，已跳过"
- **AC-03-5**：Given 目录下没有任何日志文件，When 进入阶段3，Then 暂停并提示用户"指定目录下未找到日志文件"

- **AC-05-2**：Given 识别为 sync-core-service 项目，When 进入阶段3，Then 默认搜索 `node_addon.log*`
- **AC-05-3**：Given node_addon.log 中提取到 `commit_id: {8位hash}`，When 进入步骤D，Then 截取当前分支 HEAD 前 8 位比对 → 不一致则查找远程分支
- **AC-05-4**：Given 日志中提取到 `version: {手动版本号}`（老版本格式），When 进入步骤D，Then 在 git log 中搜索匹配的 tag → 找到则标注版本 → 找不到则标注"版本号为手动维护，无法自动关联到具体 commit，请用户确认日志对应的代码版本"
- **AC-05-5**：Given 日志中既无 commit_id 也无 version，When 进入步骤D，Then 标注"日志中未找到版本信息，可能来自极早期版本或日志被截断"
- **AC-05-6**：Given 用户环境中未启用 DEBUG 日志（logging.conf 不存在或 DEBUG=FALSE），When 生成报告，Then 在报告中提示"当前环境未启用 DEBUG 日志，排查信息可能不足"

### US-06：通用项目诊断

**作为** 非 smart-sync / sync-core-service 项目的开发者，**我想要** 使用 auto-debug 诊断我的项目 Bug，**以便** 获得基本的日志分析和根因定位能力。

**验收标准**：

- **AC-06-1**：Given 当前目录不匹配任何已知项目特征，When 进入阶段2，Then 识别为通用项目，跳过 commit 比对等特定逻辑
- **AC-06-2**：Given 通用项目，When 进入阶段3，Then 搜索目录下所有 `.log` / `.txt` 文件（不限定文件名）
- **AC-06-3**：Given 通用项目，When 生成报告，Then 不包含"项目特定信息"章节

---

### US-07：用户确认与修复执行

**作为** 开发者，**我想要** 在代码修复前审核诊断报告并做出决策，**以便** 确保只有我认可的修改才会被执行。

**验收标准**：

- **AC-07-1**：Given 诊断报告已生成，When 展示给用户，Then 必须等待用户明确选择（确认修复 / 修改方案 / 仅采纳分析 / 否决）后才能继续
- **AC-07-2**：Given 用户选择"确认修复"，When 进入阶段6，Then 严格按照诊断报告中的修复方案执行代码修改
- **AC-07-3**：Given 用户选择"修改方案"，When 收到用户调整意见，Then 重新生成修复方案（阶段5的部分流程），再次展示给用户确认
- **AC-07-4**：Given 用户选择"否决"，When 收到否决指令，Then 流程终止，不做任何代码修改
- **AC-07-5**：Given 修复执行中发现修复方案与实际代码不符（如文件内容已变更、行号偏移），When 进入阶段6，Then 暂停并告知用户具体差异，不得擅自调整方案
- **AC-07-6**：Given 用户等待确认超过 60 秒未响应，When 等待超时，Then 流程挂起（不自动执行任何操作），等待用户回来后继续

---

### US-08：修复验证

**作为** 开发者，**我想要** 修复后自动验证代码的正确性，**以便** 确认修复未引入新问题。

**验收标准**：

- **AC-08-1**：Given 修复已执行且项目存在测试套件，When 进入阶段7，Then 自动运行项目测试套件
- **AC-08-2**：Given 测试全部通过，When 验证完成，Then 输出成功报告
- **AC-08-3**：Given 测试存在失败，When 验证完成，Then 输出失败详情，提示用户选择：回滚 / 继续修复 / 终止
- **AC-08-4**：Given 项目无测试套件，When 进入阶段7，Then 执行语法检查 / lint / 类型检查（按项目实际情况选择可执行的），并在报告中标注"项目无测试套件，仅执行了静态检查"
- **AC-08-5**：Given 用户选择"回滚"，When 收到回滚指令，Then 将修改的文件恢复至修复前的状态

---

### US-09：多根因分析

**作为** 开发者，**我想要** 在分析出多个可能的根因时看到完整的候选列表，**以便** 我自行判断哪个最可能。

**验收标准**：

- **AC-09-1**：Given 代码分析发现多个可能的根因，When 生成报告，Then 列出所有候选根因，每个附带独立的证据链和概率评估
- **AC-09-2**：Given 多个候选根因，When 展示报告，Then 按置信度从高到低排序
- **AC-09-3**：Given 所有候选根因的置信度均 < 50/100，When 生成报告，Then 明确标注"当前分析结果确定性较低"，并请求用户提供更多信息

---

### US-10：进度可追踪

**作为** 开发者，**我想要** 在诊断过程中随时查看当前进度，**以便** 了解 auto-debug 执行到了哪个阶段。

**验收标准**：

- **AC-10-1**：Given auto-debug 正在执行，When 任意时刻，Then 每个 Task 的状态（pending / in_progress / completed）可通过 TaskList 查看
- **AC-10-2**：Given 某阶段执行失败，When 状态更新，Then 对应 Task 的状态应反映失败（保持在 in_progress，附带失败原因描述）
- **AC-10-3**：Given 子 Agent 正在执行，When 主 Agent 等待子 Agent 返回，Then 对应 Task 显示 in_progress

---

### US-11：Bug 模式记忆

**作为** 开发者，**我想要** auto-debug 从历史诊断中积累经验，**以便** 后续遇到相似 Bug 时能更快定位。

**验收标准**：

- **AC-11-1**：Given 一次诊断成功完成（用户确认修复且验证通过），When 流程结束，Then 将 Bug 类型、错误模式、修复模式、项目上下文、置信度记录到 memory
- **AC-11-2**：Given 新 Bug 诊断时发现与历史 Bug 模式高度相似（相同项目 + 相同模块 + 相同错误模式），When 生成报告，Then 在报告中引用历史案例
- **AC-11-3**：Given 历史记忆与新分析结论冲突，When 判断时，Then 以当前分析为准，记忆仅作参考，不覆盖当前结论

## 十二、错误处理与超时策略

### 12.1 错误码体系

| 错误码 | 阶段 | 触发条件 | 恢复动作 | 用户提示 |
|--------|------|----------|----------|----------|
| E0101 | 阶段1 | Bug 描述为空或无意义 | 暂停，请求补充 | "Bug 描述不明确，请提供：1) 现象 2) 预期行为 3) 实际行为" |
| E0102 | 阶段1 | 日志输入方式无法识别 | 暂停，请求澄清 | "无法识别日志输入方式，请提供内联日志或有效的日志路径" |
| E0103 | 阶段1 | Bug 描述与日志内容矛盾 | 暂停，请求确认 | "Bug 描述与日志内容不一致：{具体矛盾点}，请确认" |
| E0201 | 阶段2 | 当前目录非项目根目录 | 暂停，通知切换 | "当前目录不是有效的项目根目录（未找到 .git / package.json 等标识文件），请切换到项目根目录后重新执行" |
| E0202 | 阶段2 | 项目代码文件不可读 | 暂停，提示权限 | "项目代码文件无法读取，请检查文件权限" |
| E0203 | 阶段2 | 日志路径不存在 | 暂停，提示路径 | "日志路径不存在：{path}，请确认路径后重新执行" |
| E0204 | 阶段2 | 日志路径不可写（无法输出提取结果） | 暂停，提示权限 | "日志路径不可写：{path}，无法保存提取结果，请检查写入权限" |
| E0301 | 阶段3 | 子 Agent 超时（>120s） | 主 Agent 评估是否自行处理或询问用户 | "日志提取超时，{超时文件列表}。是否：1) 继续等待 2) 跳过超大文件 3) 终止" |
| E0302 | 阶段3 | 日志文件损坏或不完整 | 跳过该文件，继续处理其他文件 | "日志文件 {filename} 可能已损坏，已跳过" |
| E0303 | 阶段3 | 匹配行数超限（>500行/文件 或 >1000行/总计） | 子 Agent 执行聚合摘要 | "匹配结果较多（{count}行），已自动聚合为关键事件摘要。如需查看完整提取结果，请查看：{extracted_file_path}" |
| E0304 | 阶段3 | 指定目录下无日志文件 | 暂停，提示用户 | "指定目录下未找到日志文件（.log / .txt），请确认路径" |
| E0401 | 阶段4 | 子 Agent 超时（>120s） | 主 Agent 评估是否自行处理或询问用户 | "代码分析超时，{分析进度}。是否：1) 继续等待 2) 基于当前分析结果生成报告 3) 终止" |
| E0402 | 阶段4 | 无法定位出错代码位置 | 列出已分析的信息，请求用户补充 | "无法从日志中定位到具体的出错代码位置。已找到的相关线索：{线索列表}。请提供更多信息（如具体的文件名或函数名）" |
| E0403 | 阶段4 | 分析出多个根因且置信度均 < 50 | 标注低确定性，请求用户补充信息 | "当前分析确定性较低，可能的根因有 {N} 个，最高置信度仅 {score}/100。请提供更多信息以缩小范围" |
| E0501 | 阶段5 | 报告文件写入失败 | 重试一次，仍失败则仅终端展示 | "报告文件写入失败：{path}，诊断报告仅在终端中展示，请手动保存" |
| E0601 | 阶段6 | 修复方案与实际代码不符 | 暂停，告知具体差异 | "修复方案与实际代码不一致：{差异描述}。可能原因：文件已被修改。请确认是否继续" |
| E0602 | 阶段6 | 代码修改执行失败（Edit 工具报错） | 暂停，展示错误信息 | "代码修改失败：{错误信息}。请检查文件状态后重试" |
| E0701 | 阶段7 | 测试执行失败 | 输出失败详情，等待用户决策 | "修复后测试未通过：{失败详情}。请选择：1) 回滚修改 2) 继续修复 3) 终止" |
| E0702 | 阶段7 | 回滚失败（文件状态异常） | 暂停，提示手动处理 | "自动回滚失败：{原因}。请手动检查文件：{文件列表}" |

### 12.2 超时策略

| 环节 | 最大执行时间 | 超时行为 |
|------|-------------|----------|
| 子 Agent（日志提取） | 120 秒 | 主 Agent 评估：若已有部分结果 → 继续使用部分结果；若无结果 → 询问用户 |
| 子 Agent（代码分析） | 120 秒 | 主 Agent 评估：若已有部分分析 → 基于部分结果生成报告；若无分析 → 询问用户 |
| 用户确认等待 | 无硬性超时 | 流程挂起，不自动执行任何操作，等待用户回来 |
| 全流程（阶段1-7） | 5 分钟（MVP 基线） | 若总耗时 > 5 分钟，在报告中标注"本次诊断耗时较长" |
| 单个 Grep 操作 | 30 秒 | 超时则跳过该文件 |
| 单个文件读取（代码分析） | 10 秒 | 超时则跳过该文件，在分析中标注 |

### 12.3 子 Agent 失败时的主 Agent 接管规则

| 失败类型 | 判断条件 | 接管动作 |
|----------|----------|----------|
| 超时 | 子 Agent 执行时间 > 120s | 评估已有结果：可用则继续，不可用则询问用户 |
| 空结果 | 子 Agent 返回结构化结果为空 | 主 Agent 尝试自行执行一次简化版分析；仍无结果则暂停询问用户 |
| 输出格式异常 | 子 Agent 返回结果不符合预期 schema | 主 Agent 尝试解析可用字段；无法解析则询问用户 |
| 工具异常 | 子 Agent 报告工具调用失败 | 主 Agent 记录错误，评估是否可降级处理（如换用不同工具）；不可降级则暂停询问用户 |

### 12.4 降级策略

| 降级场景 | 降级方案 |
|----------|----------|
| ripgrep (rg) 不可用 | 使用 Grep 工具（内置）替代；若 Grep 也失败，使用 Bash + grep/grep -E |
| 项目无测试套件 | 跳过测试执行，仅运行静态检查（lint / 类型检查） |
| 无静态检查工具 | 跳过自动验证，提示用户手动验证 |
| 日志路径不可写 | 提取结果和报告仅终端展示，不写入文件 |
| 子 Agent 不可用（极端情况） | 主 Agent 同步执行全流程，但需警告用户"上下文可能较大" |

## 十三、配置参数清单

### 13.1 全局配置

| 配置项 | 键名 | 默认值 | 取值范围 | 说明 |
|--------|------|--------|----------|------|

| 日志级别过滤 | `log_level_filter` | `ERROR,WARNING` | 任意日志级别组合 | 优先检索的日志级别 |
| 全流程最大耗时 | `max_total_duration` | 300 | 60-1800 | 秒，超时标注提示 |
| 单文件读取超时 | `file_read_timeout` | 10 | 5-60 | 秒，代码分析阶段单文件读取超时 |
| 单次 Grep 超时 | `grep_timeout` | 30 | 5-120 | 秒，单次 Grep 操作超时 |

### 13.2 项目特定配置

| 配置项 | 键名 | 适用项目 | 默认值 | 说明 |
|--------|------|----------|--------|------|
| 默认搜索日志文件 | `default_log_files` | smart-sync | `debug.log*,monitor.log*` | 逗号分隔的 glob 模式 |
| 默认搜索日志文件 | `default_log_files` | sync-core-service | `node_addon.log*` | 逗号分隔的 glob 模式 |
| 默认搜索日志文件 | `default_log_files` | 通用 | `*.log,*.txt` | 逗号分隔的 glob 模式 |
| 最大分析文件数 | `max_log_files` | 全部 | 50 | 目录下最多分析的日志文件数 |
| commit 比对启用 | `enable_commit_comparison` | smart-sync / sync-core-service | true | 通用项目默认 false |
| DEBUG 日志可用检查 | `check_debug_logging` | sync-core-service | true | 检查 logging.conf 是否启用 DEBUG |

### 13.3 配置来源与优先级

优先级从高到低：

1. **命令行参数**（`/auto-debug ... --keyword xxx`）
2. **项目级配置文件**（项目根目录下 `.auto-debug-config.json`）
3. **全局配置文件**（`~/.auto-debug/config.json`）
4. **内置默认值**（上表中的默认值）

### 13.4 配置文件格式

```json
{
  "log_context_lines": 5,
  "sub_agent_timeout": 120,
  "max_matches_per_file": 500,
  "max_total_matches": 1000,
  "log_level_filter": "ERROR,WARNING",
  "default_log_files": "debug.log*,monitor.log*",
  "enable_commit_comparison": true
}
```

## 十四、成功指标与性能基线

### 14.1 成功指标

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 诊断准确率 | ≥ 80%（根因在候选列表前 2 名中） | 人工评审：收集 30+ 真实 Bug 案例，由开发者判定根因是否在候选列表前 2 |
| 平均诊断耗时 | ≤ 5 分钟（从启动到报告生成） | 自动记录：每次执行记录起止时间戳 |
| 用户确认率 | ≥ 70%（用户选择"确认修复"的比例） | 自动统计：记录用户决策分布 |
| 修复验证通过率 | ≥ 85%（验证阶段通过的比例） | 自动统计：记录验证结果 |
| 用户满意度 | ≥ 4/5 | 季度问卷或使用后即时评分 |

### 14.2 性能基线（MVP 阶段）

| 场景 | 基线值 | 超出时的行为 |
|------|--------|-------------|
| 单个日志文件大小 | ≤ 100MB | 超过则跳过轮转文件，仅分析当前写入文件 |
| 日志文件总数 | ≤ 50 | 超过则仅分析最近 50 个，报告中标注 |
| 关键词匹配行数（单文件） | ≤ 500 行 | 超过则触发子 Agent 聚合摘要 |
| 关键词匹配行数（总计） | ≤ 1000 行 | 超过则触发子 Agent 聚合摘要 |
| 子 Agent 返回数据大小 | ≤ 100KB | 超过则要求子 Agent 增加摘要 |
| 源代码文件大小 | ≤ 1MB | 超过则仅读取出错位置附近 ±200 行 |
| 全流程耗时 | ≤ 5 分钟 | 超过则在报告中标注"本次诊断耗时较长" |

### 14.3 不适用的性能场景

以下场景在 MVP 阶段**不保证**性能，应提示用户：

| 场景 | 原因 | 建议 |
|------|------|------|
| 日志总大小 > 5GB | Grep 仍可在秒级完成，但上下文提取和聚合耗时显著增加 | 建议用户指定时间范围缩小搜索 |
| 项目代码文件 > 10,000 个 | 代码分析子 Agent 上下文压力大 | 建议用户指定出错模块路径 |
| 二进制/编译产物混入日志目录 | 文件发现阶段可能误匹配 | 建议用户指定具体日志文件 |
| 网络驱动器上的日志 | I/O 延迟高 | 建议用户先将日志复制到本地 |
