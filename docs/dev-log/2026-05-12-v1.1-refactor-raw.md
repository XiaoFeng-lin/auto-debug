● auto-debug 项目分析

  项目定位

  auto-debug 是一个面向 Claude Code / Codex 等 Agent 平台的 Skill（技能插件），用于自动化 Bug 诊断与修复。用户通过
  /auto-debug <Bug描述> --log <日志> 调用，系统自主完成从日志分析到代码修复的全流程。

  当前状态：MVP 文档级开发完成，尚未经过实际测试验证。

  ---
  核心架构

  采用 主-子 Agent 分层协作 模式：

  主 Agent（协调者）
    ├─ 子 Agent A: 日志提取与清洗（输入: 日志路径+关键词；输出: 结构化日志）
    ├─ 子 Agent B: 代码分析与根因定位（输入: 日志+源码；输出: 根因+修复建议）
    └─ 主 Agent: 协调调度 + 报告生成 + 修复执行

  7 阶段工作流：输入校验 → 环境检查 → 日志提取 → 代码分析 → 诊断报告 → 用户确认 → 修复验证。

  ---
  关键设计亮点

  维度: 上下文隔离
  设计: 大量日志处理由子 Agent 执行，避免污染主 Agent 上下文
  ────────────────────────────────────────
  维度: Grep-first
  设计: 绝不全文读取大日志，用 ripgrep/grep 精准检索后提取上下文
  ────────────────────────────────────────
  维度: 项目适配
  设计: 自动识别 smart-sync（Python）和 sync-core-service（C++17），支持 commit hash 比对、分支标注、DEBUG 日志检查
  ────────────────────────────────────────
  维度: 安全约束
  设计: 分析阶段只读；修复必须经用户确认；不修改原始日志
  ────────────────────────────────────────
  维度: 进度追踪
  设计: 用 TaskCreate/TaskUpdate 跟踪每阶段状态
  ────────────────────────────────────────
  维度: 错误体系
  设计: 完整的 E0101~E0702 错误码，每个错误都有明确的恢复动作
  ────────────────────────────────────────
  维度: 置信度评分
  设计: 日志证据 × 40% + 代码分析 × 35% + 复现可能 × 25%

  ────────────────────────────────────────

  ────────────────────────────────────────
  风险点: Windows 路径假设
  说明: 文档中大量 D:\ 路径示例，跨平台兼容性考虑不足

  ---
  总体评价

  设计上：这是一个架构清晰、文档完备、安全约束严格的 Skill 规范。主-子 Agent 分工、Grep-first
  策略、置信度评分、错误码体系都体现了较好的工程思维。

  实现上：它目前只是一个**"规范即实现"**的文档型 Skill，没有代码、没有测试、没有辅助脚本。实际运行效果完全取决于底层 LLM

  import platform

  def is_valid_log_path(path_str: str) -> bool:
      p = Path(path_str)
      return p.exists() and (p.is_file() or p.is_dir())

  def get_default_log_dir() -> Path:
      system = platform.system()
      if system == "Windows":
          return Path.home() / "AppData/Roaming/FangCloud"

  一、四个问题的直接回答

  3. config/default-config.json 当前起到什么作用？

  现状：它目前是一个**"人类可读的默认值参考"**，而非程序自动加载的配置。

  原因：Skill 是纯 prompt 驱动的，SKILL.md 里没有代码逻辑去读取这个 JSON 文件。它目前的实际作用是：
  - 开发文档的一部分，让开发者知道有哪些配置项
  - 安装时用户可以参考它创建 .auto-debug-config.json

  要让它真正生效，SKILL.md 的 prompt 中必须包含这样的步骤："使用 Read 工具读取项目根目录的
  .auto-debug-config.json，如果不存在则读取 ~/.auto-debug/config.json，与内置默认值合并"。

  4. docs 是否应该暴露给最终用户？

  不应该。docs/ 是开发仓库的产物（PRD、架构设计、实施计划），面向开发者和维护者。实际安装 skill 时，用户只需要：

  - SKILL.md — 核心技能（精简版）
  - references/*.md — 按需加载的项目特定知识
  - scripts/*.py — 辅助脚本（如有）
  - config/default-config.json — 配置模板

  开发文档应保留在 Git 仓库中，skill 的分发包/安装说明里只列出必要的文件。

  npx skills add 的真相

  这不是 npm install，而是 Vercel Labs 的 skills CLI 工具 (https://github.com/vercel-labs/skills)。它的工作方式是：

  npx skills add owner/repo        # 从 GitHub 仓库获取 skill
  npx skills add owner/repo --skill subdir   # 从仓库子目录获取

  和你设想的 npm 包安装完全不同：
  - 不需要 npm publish
  - 不需要 package.json 的 files 字段
  - 直接从 GitHub 拉取内容，安装到 ~/.claude/skills/ 或 .claude/skills/

  ---
  关键发现（矛盾信息）

  关于它到底复制哪些文件，我查到了两种说法：

  ┌───────────────┬───────────────────────────────────────────────────────┐
  │     来源      │                         说法                          │
  ├───────────────┼───────────────────────────────────────────────────────┤
  │ Vercel README │ "Downloads the full repo"，支持子目录安装，会递归搜索 │
  ├───────────────┼───────────────────────────────────────────────────────┤
  │ 使用教程      │ "Only fetches and copies the SKILL.md file"           │
  └───────────────┴───────────────────────────────────────────────────────┘

  这意味着存在不确定性：如果 skills CLI 只复制 SKILL.md，那我们拆分出来的 references/ 和 config/ 就不会被安装，Skill
  会失效。

● 这两个参数确实不适合放在全局 default-config.json 里，原因如下：

  ---
  default_log_files 的问题

  debug.log 和 monitor.log 是 smart-sync 特有 的日志文件名。其他项目的日志名完全不同：
  - sync-core-service → node_addon.log
  - 通用项目 → 可能叫 app.log、service.log、output.log 等

  把这个写在全局配置里，意味着非 smart-sync 项目调用时，默认搜索 debug.log*
  会什么也找不到，然后需要回退到全量扫描。这会让"默认值"失去意义，甚至产生误导。

  ---
  check_debug_logging 的问题

  这个参数是 sync-core-service 特有 的。

  作用：在环境检查阶段验证 logging.conf 中是否启用了 DEBUG 级别日志。

  启用 vs 不启用的影响：
  - 启用 DEBUG：日志会记录大量细节（函数入参、内部状态、执行路径），对于定位复杂逻辑错误、间歇性崩溃、数据流异常非常关键
  - 不启用 DEBUG：只保留
  ERROR/WARNING，某些问题（如"为什么某次调用返回了错误码"）缺少上下文，只能靠代码静态分析，诊断准确率会下降

  smart-sync 不需要这个检查，因为它的 debug.log 本身就是 DEBUG 级别，不需要额外配置。

  ---
  我的建议

  把这两个参数从全局 default-config.json 中删除，放到各自项目的 references/*.md 中：

  └─────────────────────┴───────────────────┴────────────────────────────────────────────────────┘
